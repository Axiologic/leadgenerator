# DS03: LLM Response Parsing & Document Extraction

## Problem Statement
LLM models vary widely in their ability to return clean JSON. Some return markdown-wrapped JSON, some add explanatory text, some return numbered lists instead of arrays. Search models return search results instead of following instructions at all. The system must handle all these cases gracefully.

## JSON Array Extraction Strategy (`extractJsonArray`)

The parser attempts extraction in this priority order:

### 1. Direct JSON Parse
```
Input:  [{"name":"Alice"},{"name":"Bob"}]
Result: Parsed directly via JSON.parse()
```

### 2. Markdown Code Fence Extraction
```
Input:  Here are the results:
        ```json
        [{"name":"Alice"}]
        ```
Result: Extract content between ``` fences, parse as JSON
```

### 3. Bracket Pair Extraction
```
Input:  I found these leads: [{"name":"Alice"}] Hope this helps!
Result: Find first '[' and last ']', parse substring as JSON
```

### 4. Line-by-Line Fallback (for string arrays only)
```
Input:  1. European AI startups in healthcare
        2. Horizon Europe project coordinators
        3. Nordic fintech companies
Result: Strip numbering/bullets, return as string array
```

**Safeguards for line fallback:**
- Minimum 3 lines required
- Each line must be 8-200 characters
- Lines matching `/^(example|query \d|here|sure|return)/i` are filtered out
- Prevents parroting prompt examples like "query 1", "query 2"

### 5. Failure
If none of the above work, throw an error with the first 150 chars of the raw response. This error propagates to the frontend as a visible error message.

## Model Selection Guidelines

| Model Type | Good For | Bad For |
|-----------|----------|---------|
| Chat/instruct models (copilot-gpt-4.1, etc.) | All tasks — follows JSON instructions | — |
| Search models (exa-search, Tavily-search) | Web search augmentation | JSON generation, instruction following |
| Code models (gpt-5.3-codex) | Structured output | May be overkill for simple tasks |

**Rule**: Never use search-tier models for discovery/suggest/extraction tasks. They return "Search Results for: ..." instead of JSON.

## PDF Text Extraction (`extractTextFromPDF`)

Zero-dependency extraction from PDF buffers:

1. **Primary**: Find BT (Begin Text) / ET (End Text) operators in the PDF stream
   - Extract `(string) Tj` operators (single string show)
   - Extract `[(string)(string)] TJ` operators (array show)
2. **Fallback**: If BT/ET yields nothing, extract all parenthesised strings `(...)` longer than 4 chars that contain at least 3 consecutive letters

**Limitations:**
- Cannot decompress FlateDecode streams (most modern PDFs use compression)
- Works well for uncompressed/text-based PDFs
- Scanned PDFs (image-only) will return empty text
- For production use, consider adding a dependency like `pdf-parse`

## DOCX Text Extraction (`extractTextFromDOCX`)

Zero-dependency extraction from DOCX buffers:

1. Parse ZIP local file headers (PK signature `0x04034b50`)
2. Find `word/document.xml` entry
3. If uncompressed (method 0): read directly
4. If deflated (method 8): decompress using Node.js built-in `zlib.inflateRawSync`
5. Strip XML tags, decode HTML entities, normalize whitespace
6. Convert `<w:br>` to newlines, `</w:p>` to newlines

**Limitations:**
- Only reads `word/document.xml` (ignores headers, footers, footnotes)
- Does not handle `.doc` (legacy binary format) — only `.docx`

## HTML Content Cleaning (`extractMainContent`)

Before sending HTML to LLM for lead extraction:

1. Check if content contains `<` — if not, it's already plain text (PDF/DOCX output)
2. Remove entire blocks: `<script>`, `<style>`, `<nav>`, `<footer>`, `<header>`, `<aside>`, `<noscript>`
3. Remove HTML comments `<!-- ... -->`
4. Strip all remaining HTML tags
5. Collapse whitespace to single spaces
6. Truncate to 8000 characters

**Why this matters:** Without cleaning, every page on a site produces the same leads (extracted from navigation/footer boilerplate like "CORDIS", "European Commission") instead of page-specific content.

## Caching Strategy

| Content Type | Cache Location | Cache Key | Storage Format |
|-------------|---------------|-----------|----------------|
| HTML pages | `data/cache/{hash}.html` | MD5(url) | UTF-8 text |
| PDF files | `data/cache/{hash}.pdf` | MD5(url) | Binary buffer |
| DOCX files | `data/cache/{hash}.docx` | MD5(url) | Binary buffer |

- Cache is permanent (no TTL) — manual deletion required to re-fetch
- `data/` directory is in `.gitignore`
- Text extraction happens after cache read (not cached separately)
