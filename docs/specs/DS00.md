# DS00: Lead Generator and Manager Specifications

## Vision and Problem Statement
The goal is to create a lightweight, dependency-free web application for generating and managing marketing leads. The application will leverage LLMs (via AchillesAgentLib) and web scraping to discover potential leads (persons or organizations), gather contact information (email, LinkedIn), and provide a management interface for scoring, contacting, and note-taking.

## Intended Users and Context of Use
Marketing professionals or business owners who need a simple, self-hosted tool to discover and manage leads without complex CRM overhead or recurring subscription costs.

## Functional Requirements

### 1. Authentication
- Simple password-based authentication.
- Ability to change the password via the interface.
- Session persistence using a simple cookie or token stored in the browser.

### 2. Lead Discovery (Marketing Agent)
- Discover persons or organizations based on a topic.
- Use LLM search models for broad discovery.
- **Recursive Scraper:**
    - Input: Starting URL.
    - Logic: Fetch page, extract internal links, and visit them recursively up to a configurable depth.
    - Extraction: Use LLM to identify and extract lead information (name, organization, email, LinkedIn) from page content.
    - Caching: Local cache for visited URLs and analysis results to prevent redundant processing.
- Target specific sites (e.g., CORDIS) for deep document search.
- Extract contact details: LinkedIn profiles and email addresses.
- Deduplication: Skip already analyzed sites or search results using a local cache.

### 3. Configuration & Settings
- Simple password-based authentication and management.
- **LLM Provider Configuration:**
    - Interface to select specific LLM models or tiers (fast/deep) for different application tasks:
        - **Discovery:** Finding initial candidates/sites.
        - **Extraction:** Parsing content for lead details.
        - **Scoring:** Evaluating lead quality.
    - Integration with `AchillesAgentLib` to list available providers and models.
- **Scoring Configuration:** Define criteria for automated lead prioritization.

### 4. Lead Management
- Store leads in a local JSON "database".
- Prioritize leads based on configurable scoring criteria.
- Mark leads as "Not Interested".
- Mark leads as "Contacted".
- Add and save notes for contacted leads.

### 5. Persistence
- All data (leads, cache, configuration) stored in the `data/` folder.
- `data/` folder excluded from Git.

## Technical Specifications

### Server
- Node.js `http` module (no Express, no dependencies).
- File-based persistence using `fs/promises`.
- Integration with `../AchillesAgentLib` for LLM operations.

### Frontend
- Single-page application (SPA) style.
- Vanilla HTML5, CSS3, and ES6+ JavaScript.
- Elegant and responsive design.

### Data Model
- `data/leads.json`: Array of lead objects `{ id, name, organization, topic, source, email, linkedin, score, status, notes, createdAt }`.
- `data/cache.json`: Map of analyzed URLs or search queries to avoid repetition.
- `data/config.json`: Store hashed password, scoring rules, and task-specific LLM settings.

## Implementation Plan

### Phase 1: Foundation
1. Create directory structure: `src/`, `src/public/`, `data/`, `docs/specs/`.
2. Add `data/` to `.gitignore`.
3. Implement `src/storage.mjs`: CRUD operations for JSON files.
4. Implement `src/server.mjs`: Basic HTTP server for static files and API endpoints.

### Phase 2: Authentication & Config
1. Implement password check middleware/logic.
2. Create settings page for password change, scoring configuration, and **LLM Task configuration**.
3. Integrate with `AchillesAgentLib` to fetch available models.

### Phase 3: Recursive Scraper & Discovery Logic
1. Implement `src/marketingAgent.mjs`:
   - Integration with `AchillesAgentLib` (LLMAgent).
   - **Recursive Scraping Logic:** Fetch pages, extract links, visit children with depth control.
   - **Lead Extraction:** LLM-powered identification of person/org, email, and LinkedIn from page content.
   - **Caching Logic:** Save analysis to `data/cache.json` to prevent re-scraping same URLs.
2. Create API endpoints for starting a discovery task or a recursive scrape task.

### Phase 4: Frontend Implementation
1. `index.html`: Main layout with sections for Discovery, Scraper, Lead List, and Settings.
2. `style.css`: Clean, professional UI.
3. `app.js`: Handle API calls, state management, and DOM updates.

### Phase 5: Lead Management & Scoring
1. Implement lead prioritization logic based on scoring config.
2. Add notes and status update functionality.
3. Final UI/UX polish.

## Verification Plan
- Manual testing of the discovery process with various topics.
- Verify data persistence after server restart.
- Verify deduplication by running the same search twice.
- Test password change and authentication lockout.
